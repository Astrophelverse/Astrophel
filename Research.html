// JUP-H-03 · Jupiter Lend · Code4rena 2026-02 Absorption Inflation — Fund Drainage via rebalance() programs/vaults/src/module/user.rs · Rust / Anchor / Solana
High / Critical Submitted Code4rena

When a branch is absorbed into the vault, total_borrow in VaultState is never decremented. The absorbed debt silently disappears from tick/branch accounting, creating phantom debt that rebalance() treats as real profit and drains from the Liquidity Layer. Unlike liquidate() which correctly calls reduce_total_borrow(), absorb() never does — a single missing line that enables continuous fund extraction.

// Root Cause
; user.rs — absorb() — L1112-1115
vault_state.add_absorbed_col_amount(col_absorbed)?;    // tracked ✓
vault_state.add_absorbed_debt_amount(debt_absorbed)?;  // tracked ✓
// vault_state.reduce_total_borrow(debt_absorbed)      // MISSING ✗
Ok((col_absorbed.cast()?, debt_absorbed.cast()?))

; liquidate() does it correctly:
vault_state.reduce_total_supply(current_data.total_col_liq)?;
vault_state.reduce_total_borrow(current_data.total_debt_liq)?; // done here ✓
// absorb() never calls this                                    // BUG ✗
// Impact
💸
Active Fund Drainage
rebalance() reads inflated total_borrow and drains the delta from the Liquidity Layer to the rebalancer address as "profit".
📈
Compounds Forever
Every absorption event inflates total_borrow permanently. Absorption is a core protocol feature — not an edge case.
🏦
Lender Deposits at Risk
Real user deposits in the Liquidity Layer are the source of the phantom drain. Full insolvency is the end state.

Add one line to absorb() after line 1113:

// JUP-H-02 · Jupiter Lend · Code4rena 2026-02 Net/Raw Debt Mismatch — Liquidation DoS via Underflow programs/vaults/src/module/user.rs · Rust / Anchor / Solana
High Submitted Code4rena

VaultState.total_borrow tracks net debt — raw debt minus tick-rounding dust. During liquidation, reduce_total_borrow() is called with the raw debt amount. This causes a subtraction underflow in safe_sub(), reverting the transaction and permanently blocking all future liquidations for that vault. Since dust is inherent to tick-math, this affects every single borrow in the protocol.

// The Math
total_borrow  = raw_debt - dust  =>  666,667   // NET — what VaultState stores
total_debt_liq = raw_debt        =>  1,000,000  // RAW — what liquidate() passes

safe_sub(666_667, 1_000_000) => UNDERFLOW => REVERT  // ✗
// Proof of Concept
// cargo test --test net_raw_mismatch_poc -p library -- --nocapture

total_borrow (NET): 666667
subtract amount (RAW): 1000000
Result: None  // underflow confirmed

IMPACT: All liquidations halt permanently. Bad debt accumulates.
test test_liquidation_underflow_revert ... ok. 1 passed; 0 failed
// Fix
Subtract only the net portion, or unify accounting basis:

- vault_state.reduce_total_borrow(current_data.total_debt_liq)?;
+ vault_state.reduce_total_borrow(current_data.total_debt_liq.safe_sub(dust)?)?;
Submitted March 2026 Awaiting judge validation
